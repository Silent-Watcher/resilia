# Contributing Guidelines

Thank you for considering contributing to this project. Contributions of all kinds are welcome, including bug reports, feature requests, documentation improvements, and code changes.

Please take a moment to review this document to ensure a smooth and effective contribution process.

---

## Ways to Contribute

You can contribute in several ways:

* Reporting bugs
* Suggesting new features or improvements
* Improving documentation
* Refactoring or optimizing existing code
* Adding tests
* Reviewing pull requests

No contribution is too small.

---

## Getting Started

1. Fork the repository.
2. Clone your fork locally.
3. Create a new branch for your change:

   ```bash
   git checkout -b feature/your-feature-name
   ```
4. Make your changes.
5. Commit with a clear and descriptive message.
6. Push your branch and open a Pull Request.

---

## Issues

### Reporting Bugs

When opening a bug report, please include:

* A clear and descriptive title
* Steps to reproduce the issue
* Expected behavior
* Actual behavior
* Environment details (OS, Node version, etc.), if relevant

### Feature Requests

Feature requests should include:

* The problem you are trying to solve
* A proposed solution or approach
* Any alternatives you considered

---

## Pull Requests

Before submitting a pull request, please ensure:

* The PR addresses a single concern (bug fix, feature, refactor, etc.)
* Code is readable and well-structured
* Existing tests pass
* New functionality includes tests where appropriate
* Documentation is updated if behavior changes

### PR Guidelines

* Keep pull requests focused and small when possible
* Reference related issues (e.g. `Closes #12`)
* Avoid unrelated formatting or stylistic changes

---

## Code Style

* Follow the existing code style and conventions
* Prefer clarity over cleverness
* Write meaningful variable and function names
* Keep functions small and focused

If a linter or formatter is configured, please ensure it passes before submitting.

---

## Commit Messages

* Use clear, concise commit messages
* Prefer present tense: “Add validation for input”
* Avoid vague messages like “fix stuff” or “updates”

---

## Tests

* Add tests for new features and bug fixes when applicable
* Ensure all tests pass before submitting a PR
* Do not reduce test coverage without justification

---

## Documentation

* Update README or related docs if behavior changes
* Document public APIs and non-obvious logic
* Keep documentation concise and accurate

---

## Code of Conduct

By participating in this project, you agree to follow the project’s Code of Conduct. Be respectful, constructive, and professional in all interactions.

---

## Questions

If you are unsure about anything, feel free to open an issue or start a discussion before working on a change.

We appreciate your contribution and your time.
